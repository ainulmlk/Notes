AstuceRègle rapide : 1er octet < 128 → A ; 128–191 → B ; 192–223 → C ; 224+ → D/E.
  • Adresse réseau : tous les bits hôte à 0 (ex. 192.168.1.0). Identifie le réseau.
  • Adresse de diffusion (broadcast) : tous les bits hôte à 1 (ex. 192.168.1.255). Envoie à tous les hôtes.
  • Loopback : 127.0.0.0/8 (typiquement 127.0.0.1) — ne quitte jamais la machine.
  • APIPA : 169.254.0.0/16 — auto-configuration quand le DHCP échoue.

Ces adresses ne sont pas routées sur Internet. Elles sont utilisées en interne avec le NAT pour économiser les adresses IPv4 publiques (RFC 1918).

Le subnetting consiste à emprunter des bits hôtes pour créer des sous-réseaux à partir d'un réseau plus grand. On allonge le masque (ex. /24 → /26) pour diviser l'espace d'adressage.

Formules clés
Nombre de sous-réseaux = 2ˢ (s = bits empruntés)
Hôtes utilisables / sous-réseau = 2ʰ − 2 (h = bits hôtes restants)
Incrément = 256 − valeur octet masque
Exemple /26Masque 255.255.255.192 → 2² = 4 sous-réseaux, 2⁶−2 = 62 hôtes chacun, incrément 64.

Le supernetting (ou agrégation / summarization) consiste à regrouper plusieurs réseaux contigus en un réseau plus grand en raccourcissant le masque. Utilisé dans BGP et OSPF pour réduire la taille des tables de routage.

Exemple192.168.0.0/24 + 192.168.1.0/24 + 192.168.2.0/24 + 192.168.3.0/24 → 192.168.0.0/22

Le VLSM (Variable Length Subnet Mask) permet d'utiliser des masques de longueurs différentes dans le même réseau. Les protocoles de routage doivent être classless (sans classe), c'est-à-dire qu'ils transportent le masque dans leurs mises à jour :

  • Compatibles VLSM : RIPv2, OSPF, EIGRP, BGP, IS-IS
  • Incompatibles : RIPv1, IGRP (classful, n'envoient pas le masque)

Une liaison point-à-point (WAN série) n'a besoin que de 2 adresses hôtes, donc le préfixe optimal est /30 (masque 255.255.255.252).

/30 – liaison point à point
2⁸⁻³⁰ − 2 = 2³² − ³⁰ = 2² − 2 = 2 hôtes utilisables ✓

TD2 — Traduction d'adresse NAT / PAT

NAT statique · NAT dynamique · PAT · Masquerading

I-b – Questions de cours Préparation

Le NAT (Network Address Translation) a été développé pour pallier l'épuisement des adresses IPv4 publiques. Il permet à plusieurs hôtes utilisant des adresses privées (RFC 1918) de partager une ou plusieurs adresses publiques pour accéder à Internet. Il apporte aussi un bénéfice secondaire de sécurité en cachant la topologie interne.

Le NAT statique établit une correspondance permanente et bijective entre une adresse privée et une adresse publique. La table de traduction est configurée manuellement.

ip nat inside source static 192.168.1.10 200.1.1.10

Usage typique : rendre un serveur interne (web, FTP) accessible depuis Internet.

interface Fa0/0
  ip nat inside    ← réseau privé (LAN)

interface Serial0/0
  ip nat outside   ← réseau public (Internet)
Règle mnémotechnique"inside" = côté LAN privé / "outside" = côté Internet public.
TermeDescription
Inside LocalAdresse privée réelle de l'hôte interne (ex. 192.168.1.5)
Inside GlobalAdresse publique qui représente l'hôte interne vu depuis Internet
Outside LocalAdresse du serveur externe vue depuis le réseau interne
Outside GlobalAdresse réelle du serveur externe (même chose en général)

Le PAT (Port Address Translation) ou NAT Overload permet à plusieurs hôtes de partager une seule adresse publique en distinguant les sessions grâce aux numéros de port TCP/UDP.

Le port mapping (ou port forwarding) est l'opération inverse : rediriger un port entrant (ex. port 80 de l'IP publique) vers un serveur interne.

ip nat inside source list 1 interface Serial0/0 overload
  • Consomme 1 adresse publique par serveur interne exposé → peu scalable.
  • Certains protocoles qui intègrent l'adresse IP dans leur payload (FTP actif, H.323, SIP) peuvent être cassés par le NAT.
  • Difficulté de tracer les connexions (débogage).

Le NAT dynamique utilise un pool (groupe) d'adresses publiques attribuées temporairement aux hôtes internes. La correspondance est créée à la demande et supprimée après timeout.

ip nat pool MON_POOL 201.200.200.17 201.200.200.18 netmask 255.255.255.252
access-list 1 permit 192.168.3.0 0.0.0.255
ip nat inside source list 1 pool MON_POOL

Le NAT Masquerading est un PAT qui utilise l'adresse IP de l'interface WAN du routeur automatiquement, sans pool fixe. Idéal quand l'IP publique est attribuée dynamiquement par le FAI (DHCP).

ip nat inside source list 1 interface Serial0/0 overload

Lorsqu'un serveur interne doit être accessible depuis Internet alors que seul le masquerading est en place. Le routeur ne peut pas deviner quel serveur interne correspond à une connexion entrante → il faut lui indiquer explicitement vers quel port/IP interne rediriger.

ip nat inside source static tcp 10.0.0.1 21 200.20.2.5 21

TD3 — Sécurisation par filtrage ACL

ACL standard · étendue · nommée · wildcard mask

I-b – Questions de cours Préparation

  • Contrôler le trafic réseau entrant/sortant sur une interface
  • Sécuriser l'accès aux ressources (serveurs, zones DMZ)
  • Filtrer selon IP source/destination, protocole, ports
  • Classifier le trafic pour QoS
  • Restreindre l'accès au routeur lui-même (lignes VTY)

Le masque générique est l'inverse du masque réseau :

Wildcard = 255.255.255.255 − Masque réseau

Exemple : /24 → 255.255.255.0 → wildcard = 0.0.0.255
Exemple : /26 → 255.255.255.192 → wildcard = 0.0.0.63
  • Bit 0 → ce bit DOIT correspondre (bit "contrôlé")
  • Bit 1 → ce bit est ignoré ("don't care")
  • host 10.1.1.1 = 10.1.1.1 0.0.0.0
  • any = 0.0.0.0 255.255.255.255

Les ACL standard filtrent uniquement sur l'IP source. Elles doivent être placées le plus proche possible de la destination (pour ne pas bloquer du trafic légitime vers d'autres destinations).

access-list [1-99] {permit|deny} source [wildcard]
! Exemples :
access-list 1 deny 192.168.3.0 0.0.0.255
access-list 1 permit any

interface Fa0/1
  ip access-group 1 out
Implicit denyToute ACL se termine par un "deny any" implicite. Toujours ajouter permit any si nécessaire.

Les ACL étendues filtrent sur IP source, IP destination, protocole, ports TCP/UDP. Elles doivent être placées le plus proche possible de la source (pour éviter trafic inutile sur le réseau).

access-list [100-199] {permit|deny} protocole source s_wild dest d_wild [eq port]

! Exemples :
access-list 110 deny tcp 10.10.10.0 0.0.0.255 host 10.1.1.10 eq 80
access-list 110 deny tcp 10.10.10.0 0.0.0.255 host 10.1.1.10 eq 443
access-list 110 permit ip any any

interface Fa0/0
  ip access-group 110 in
  • Nom descriptif plutôt qu'un numéro → meilleure lisibilité
  • Possibilité de supprimer une règle précise (impossible avec ACL numérotées en IOS classique)
  • Ajout de règles dans un ordre précis avec les numéros de séquence
ip access-list extended FILTRE_WEB
  10 deny tcp 10.10.10.0 0.0.0.255 host 10.1.1.10 eq 80
  20 permit ip any any
  no 10              ← supprime la règle 10
  15 deny udp ...    ← insère entre 10 et 20

Le wildcard 0.0.0.1 (bit 0 du dernier octet = don't care) sélectionne les adresses paires.
Le wildcard 0.0.0.0 avec .1 donne les impaires avec un masque astucieux.

Impaires : source .4 wildcard 0.0.0.2 → ne match que .5, .7 etc.

Impair = 0x01 dans le bit 0 → source wildcard 0.0.0.254 avec adresse .1
Pair = 0x00 dans le bit 0 → source wildcard 0.0.0.254 avec adresse .0

Réseau : RA – RB – RC – RD – RE avec coûts IEEE (Ethernet=100, FastEthernet=19, Gigabit=4).

PrincipeChaque routeur informe ses voisins par diffusion. Un routeur met à jour sa table si une route reçue est MEILLEURE (coût inférieur) ou NOUVELLE. L'algorithme converge quand aucune table ne change plus.

Itération 0 – Tables initiales : chaque routeur ne connaît que ses réseaux directement connectés.

Itération 1 : chaque routeur reçoit les tables de ses voisins directs et ajoute leurs routes avec son coût de liaison.

Itération 2 : propagation à 2 sauts → convergence sur ce réseau à 5 nœuds.

Coût vers destination = coût interface locale + coût annoncé par le voisin
Problème RIPv1RIPv1 envoie toute la table toutes les 30 secondes à 255.255.255.255. Risque de "count to infinity" en cas de panne. Solutions : split horizon, poison reverse, hold-down timers.
Coût OSPF = 10⁸ / bande_passante (en bps)

Ethernet (10M) → 100/1 = 10 ← coût IEEE
FastEthernet → 100/10 ≈ 10 (ou 19 en IEEE)
Gigabit (1G) → 100/1000 = 0.1 → arrondi à 1
10 Gigabit → arrondi à 1

Processus SPF :

  1. Construction de la base topologique (LSDB) commune à tous les routeurs
  2. Chaque routeur calcule l'arbre des plus courts chemins depuis lui-même (SPF tree)
  3. Les routes sont déduites de cet arbre → table de routage
Cumul des coûtsOn additionne les coûts des interfaces émettrices (celles par lesquelles le paquet PART). Le coût d'arrivée sur une interface n'est pas compté.

  network 192.168.4.0 0.0.0.255 area 0
  passive-interface Fa0/0   ← ne pas envoyer d'OSPF sur réseau local

Redistribution OSPF → RIP sur Rb :

router rip
  redistribute ospf 1 metric 5   ← injecter routes OSPF dans RIP avec métrique 5
ProblèmeLa redistribution OSPF → RIP peut créer des boucles si RIP redistribue aussi vers OSPF. La table de Rb peut avoir des routes en double avec des métriques incohérentes. Solution : filtrer avec des route-maps ou des distribute-lists.

TD5 — Système d'adressage IPv6

Format · Types d'adresses · EUI-64 · Plan d'adressage

I-b – Questions de cours Préparation

Une adresse IPv6 = 128 bits = 8 groupes de 16 bits (hextets) séparés par :

Règle 1 : Supprimer les zéros non significatifs dans chaque hextet
0db8 → db8 | 0000 → 0

Règle 2 : Remplacer UNE seule séquence consécutive de hextets nuls par ::

:: ne peut apparaître qu'une seule foisSinon l'adresse devient ambiguë.
TypeDescriptionPréfixe
UnicastUn seul destinataireGUA 2000::/3, LLA fe80::/10, ULA fc00::/7
MulticastGroupe de destinatairesff00::/8
AnycastLe plus proche d'un groupe(même format unicast, marqué anycast)
Pas de broadcast en IPv6 !Remplacé par le multicast solicited-node ff02::1:ffxx:xxxx.

Préfixe : fe80::/10. Générée automatiquement sur chaque interface IPv6 à partir de l'adresse MAC (EUI-64).

  • Valable uniquement sur le lien local (non routable)
  • Utilisée pour : découverte de voisins (NDP), adresse de passerelle par défaut, protocoles de routage (RIPng, OSPFv3)
  • Chaque interface en a une automatiquement sans configuration
MAC = AA:BB:CC:DD:EE:FF (48 bits)

1. Insérer FF:FE au milieu :
AA:BB:CC:FF:FE:DD:EE:FF

2. Inverser le bit U/L (7ème bit du 1er octet) :
AA → A8 (si bit7=0, mettre à 1 ; si =1, mettre à 0)

3. Convertir en hextets IPv6 :
A8BB:CCFF:FEDD:EEFF

LLA résultante : fe80::A8BB:CCFF:FEDD:EEFF

4 sites × 8 bâtiments × 5 niveaux = 160 réseaux maximum.

Préfixe alloué : 2001:660:acad:200::/56
Bits disponibles pour le découpage : 128 − 56 = 72 bits hôtes
Mais on découpe le préfixe de réseau → utiliser les bits 57 à 64

Structure choisie (bits 57–64 = 8 bits = 2 octets en hexa) :
Bits 57–58 : Site (4 sites = 2 bits)
Bits 59–61 : Bâtiment (8 bâtiments = 3 bits)
Bits 62–64 : Niveau (5 niveaux → 3 bits = 8 valeurs, on utilise 1–5)

Chaque réseau = /64 (64 bits hôtes pour EUI-64 des hôtes)

I-c – Exercices configuration IPv6 Application

  • Pourquoi ont-ils déjà une IPv6 ? Les interfaces génèrent automatiquement une LLA (fe80::) à partir de leur MAC via EUI-64 dès qu'IPv6 est activé dans l'OS.
  • Quel type ? LLA (Link-Local Address), cohérent avec la MAC via EUI-64.
  • Connectivité IPv6 entre hôtes du même LAN ? OUI — la LLA suffit pour communiquer sur le même lien.
  • Sans IPv4 ? La LLA fonctionne toujours pour le trafic local.
  • En DHCPv4 uniquement ? Les hôtes obtiennent 169.254.x.x (APIPA) en IPv4 → pas de connectivité IPv4. IPv6 LLA toujours active.
! Sur Ra :
ipv6 unicast-routing

interface Fa0/0
  ipv6 address 2001:DB8:10::FF/64
  ipv6 enable
  no shutdown

interface Fa0/1
  ipv6 address 2001:DB8:20::FF/64
  ipv6 enable
  no shutdown

Mode automatique sur les hôtes (SLAAC) : Les hôtes reçoivent les Router Advertisements du routeur et construisent leur GUA en combinant le préfixe annoncé + EUI-64 de leur MAC.

Gateway = LLA du routeur → le test de connectivité entre les deux réseaux locaux FONCTIONNE avec la GUA grâce au routage IPv6.

Si gateway = GUAFonctionne aussi mais moins robuste. Si gateway = LLA, le paquet sort toujours par la bonne interface.
! Syntaxe route statique IPv6 :
ipv6 route prefixe/longueur {next-hop-LLA [interface] | interface}

! Exemple sur RT :
ipv6 route 2001:DB8:1::/64 2001:DB8:2::1   ← via Ra
ipv6 route ::/0 Serial0/0                   ← route par défaut

! Route par défaut (supernet de tout) :
ipv6 route ::/0 {next-hop}
Avec LLA comme next-hopIl faut spécifier l'interface : ipv6 route 2001:db8::/64 Fa0/0 fe80::1

II – RIPng et OSPFv3 TD

! Sur chaque routeur :
ipv6 router rip RIPng

! Sur chaque interface à annoncer :
interface Fa0/1
  ipv6 rip RIPng enable

! Passive interface (ne pas envoyer d'annonces sur le LAN) :
ipv6 router rip RIPng
  passive-interface Fa0/0

RIPng utilise le multicast ff02::9 pour envoyer ses mises à jour.

! Global :
ipv6 router ospf 1
  router-id 3.3.3.3   ← obligatoire ! format IPv4

! Sur chaque interface :
interface Fa0/1
  ipv6 ospf 1 area 0

! Passive interface :
ipv6 router ospf 1
  passive-interface Fa0/0
Sans router-idOSPFv3 ne peut pas démarrer s'il n'y a pas d'adresse IPv4 sur le routeur. Toujours définir le router-id manuellement.
! Sur Rb (qui tourne les deux protocoles) :

! Redistribuer OSPF dans RIP :
ipv6 router rip RIPng
  redistribute ospf 1 metric 5

! Redistribuer RIP dans OSPF :
ipv6 router ospf 1
  redistribute rip RIPng

La redistribution bidirectionnelle risque de créer des boucles de routage. Utiliser des route-maps ou des métriques appropriées pour l'éviter.
